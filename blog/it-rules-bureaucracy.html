Why I am starting with HTML: IT rules of a public bureaucracy

Mar 2026

The three vibe coding creations have all been done under HTML because:

  1. IT rules at my work are way too strict, and
  2. the technical challenges required to solve work problems are not that complex.

Ask AI chatbots on where to do vibe coding, they suggest some form of cloud platform. Wanting to keep things free, my first vibe coding attempt (RTS, soon on this site) was done on Google platforms (Sheets, Script, and Web Apps). However, using apps not approved by work or having data live outside of the work servers are big no-nos from our IT gatekeepers.

To give a bit of color:

  • We don't have MS Teams on our work phone because all work apps have to live in the Blackberry ecosystem.
  • I once asked to get access to PowerBI, but after a few lengthy rejections and a suggestion that I should use the SAP dashboard, my request was sucked into a change management blackhole.
  • Our department did a demand and feature identification exercise to procure a project management software, but was pretty quickly shut down by the overlords. MS Outlook remains the best project management tool we have.

But then again, my line of work does not require sophisticated tools. Work is mainly tracking who has approved what, where is it documented, which colleagues are informed of the various stages, etc. Complexities instead arise from work idiosyncrasies and personalities, unfortunately something and some people we have not automated yet.

Hence, HTML. It lives as an offline file that does not require contacting IT. After creating my personal dashboard (link) on HTML wihtin an hour, I wondered if HTML can be used for the more complex RTS, thereby getting off Google platforms. Claude initially gave big thumbs up to this suggestion.

Writing about it now, it feels quite banal, but I was quite excited about it when I came to this realization. Any digital initiatives at work are quickly discouraged or dismissed due to the prospect of facing the IT overlords, but with HTML I feel a lot more confident that using AI at a bureaucratic workplace like mine can be possible.

Then again, in writing this blog post I did ask Claude again if HTML is the right fit for the RTS, to which it replied:

The honest comparison: a React or Vue app would handle this architecture comfortably because those tools are built around exactly this kind of problem — state management, component re-rendering, data persistence. RTS is doing a DIY version of all of that inside a format that was originally designed for documents. It works, and it's the right call given your constraints. But you're essentially building a car engine inside a bicycle frame. It fits, it runs, but you feel the tension.

So my second conclusion here is that AI as a coworker is also a gaslighter.
