HTML, not so great

April 2026

In an earlier blog post I wrote as if I have perhaps stumbled upon html as my savior to vibe coding structure, specifically how as it can enable me to create tools purely offline. I finished my first big vibe coding project RTS and ended up paring it down significantly.

Initially, I wanted it to be a tool used by multiple colleagues, who could click on check marks on tasks that they had to do. This required, however:

  • Everyone to be looking at the same data. But html lives in a browser, and cannot connect to an offline file. Online file already discounted due to IT rules.
  • One workaround is to have a data file (JSON) in the same folder, and have the user upload it each time. Having a user, especially a secondary user who will use the tool for one out of 30 tick marks, requires frictionless access, and this workaround is sticky friction.
  • Browser based tool also relies heavily on localstorage, with each user using their own computer. Therefore, it is likely that users are not looking at the same data. This brings us back to the importance of an online database to act as the single source of truth (no not that Truth). One workaround was: at the end of the session, "save data" button will hardcode most recent data inside the html file. But for same reason as above, this means that the browser will download a new html file, which will have to replace the existing one, which requires a few renamings since the file name should be the same.
