PII Cleaner

March 2026

Germans take data security pretty seriously. I couldn't find a good meme on it so here's a joke that Gemini wrote:

A German is stranded on a deserted island. After three days, he finally sees a rescue ship. The captain shouts through a megaphone: "We're here to save you! What's your name?" The German shouts back: "I cannot tell you that without a signed data processing agreement (AVV)!"

I don't think the joke is very good, but it was the best out of five options it provided.

If Germans take data security pretty seriously, German workplaces take it even more seriously. This is problematic in using online tools like Google Translate, let alone all sorts of AI chatbots.

It would be nice if I could just use an AI chatbot to remove all Personally Identifiable Information (PII) by using a prompt like:

"Please review the following text and remove or replace all personally identifiable information (PII). This includes names, phone numbers, email addresses, physical addresses, and specific ID numbers (like SSNs or account numbers). Replace the sensitive data with descriptive placeholders like [NAME], [EMAIL], or [PHONE]. Once redacted, provide the cleaned version of the text below."

But this itself is not compliant since it requires inputting PIIs.

So I resorted to vibe coding a tool that will do this offline. Intuitively, I took the prompt and turned it around, asked Claude to create an HTML file that would do this.

What it detects

Names — People's first names, last names, and full names. The tool recognises over 1,500 common names across English, German, French, Spanish, Portuguese, Italian, Dutch, Scandinavian, Polish, Turkish, Arabic, South Asian, and African naming conventions. It also catches names written in ALL CAPS, hyphenated names like Jean-Pierre or Müller-Schmidt, and names that appear after polite openers like "Dear" or "Sehr geehrte Frau". If a name appears next to a phone number or email address, it is removed automatically. If it's less certain, it flags it for review.

Company names — Legal entity names followed by a suffix like GmbH, AG, Ltd, LLC, SARL, S.A.R.L., BV, SAS, and their variants in any capitalisation. Also catches the company name sitting above a street address in a letterhead block.

Addresses — Street addresses in German (including compound forms like Kaemmergasse, Hauptstraße, Georgstr.), English, French, Spanish, and Dutch. Also catches PO Box and Postfach entries.

Email addresses — Any standard email address format.

Phone numbers — Numbers starting with +, 00, or 0, covering international and local German and European formats. Financial figures like salaries and amounts are excluded.

URLs — Any web address beginning with http or https.

Postcodes — German five-digit postcodes and UK postcode formats.

Bank account numbers (IBAN) — All European IBAN formats, with or without spaces between digit groups.

BIC / SWIFT codes — Bank identifier codes used in international transfers.

Tax and social security numbers — German Steuernummer (slash format), Steuer-ID (11 digits), Sozialversicherungsnummer. Also Spanish DNI/NIE, Dutch BSN, and Italian Codice Fiscale.

Dates — Dates in numeric and written formats. Contract clause numbers like §4.2.1 are excluded.

IP addresses — Any IPv4 address.

What it can miss

Uncommon or unusual names — If a name doesn't appear in the built-in lists and isn't near other identifying information like an email or phone number, it may not be caught. Very rare surnames, newly invented names, or names from languages not covered (e.g. Vietnamese, Thai, Japanese given names) are the most likely to slip through.

Single names without context — A surname appearing alone mid-paragraph — "contact Müller directly" — with no title, salutation, or nearby PII may not be flagged.

City and place names — Bonn, Berlin, Frankfurt are not removed. They appear too frequently in legitimate business context to redact automatically. If a city is part of an address it will be caught, but standalone mentions will not.

Company names without legal suffixes — If a company name doesn't end in GmbH, Ltd, SARL etc. and isn't in a letterhead position above a street address, it won't be detected.

Salary figures and financial data — Numbers with decimal separators are deliberately excluded to avoid false positives on phone numbers. Amounts like 6.412,00 EUR pass through untouched. If salary figures are sensitive in your document, remove them manually.

Document-specific ID numbers — Reference numbers, case numbers, contract numbers, and internal codes with custom formats the tool doesn't recognise will not be caught.

Names embedded in sentences without capitalisation — All detection relies on capitalisation patterns. A name typed in all lowercase will not be detected.

Sensitive content that isn't PII — The tool removes identifying information, not confidential content. Commercially sensitive figures, strategic plans, internal policies, or personal opinions are outside its scope entirely.

What about ambiguous cases?

The tool is by nature conservative, so two consecutive words that start with capital letters can be recognized as a name, such as Product Information. Given that this is an HTML tool, the user has the option to decide on these edge cases — redact or keep.

There are also names that should always be redacted, but the tool does not pick up so well. Company names are often the case. Hence there is a function to explicitly write which words or names to always redact.
